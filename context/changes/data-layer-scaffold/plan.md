# Data Layer Scaffold Implementation Plan

## Overview

Install SQLModel + asyncpg + Alembic, create a shared DB connector module, define four SQLModel table models covering the full MVP schema, and generate + verify the initial Alembic migration. A smoke test confirms the connection and migration are healthy before any feature slice builds on top.

## Current State Analysis

- `backend/pyproject.toml` has only `fastapi[standard]` â€” no ORM, no driver, no migration tool
- `backend/app/main.py` is a FastAPI factory with CORS middleware and no DB lifespan (`main.py:10-29`)
- `backend/app/v1/medicines/`, `auth/`, `health/` exist as stubs with no models or DB access
- No `backend/tests/` directory exists yet (AGENTS.md: "place tests there")
- No Alembic configuration anywhere in the repo

## Desired End State

- `uv run uvicorn app.main:app --reload` starts without error and logs a successful DB connection
- `uv run alembic upgrade head` applies the initial migration cleanly against a Supabase PostgreSQL DB
- `uv run pytest` passes the smoke test confirming session acquisition works
- Four tables exist in the DB: `users`, `medication_registry`, `cabinet_entries`, `user_preferences`
- A GIN index on `medication_registry.search_vector` (tsvector) is in place for autocomplete

### Key Discoveries

- `backend/app/v1/router.py` imports all domain routers â€” new domains (`cabinet/`, `users/`) must be registered here (`router.py:1-11`)
- AGENTS.md mandates `backend/app/v1/<domain>/models.py` as the location for SQLModel table models
- asyncpg requires the connection string scheme `postgresql+asyncpg://` â€” the standard `postgresql://` used by psycopg2 will fail silently or raise at startup
- Alembic's async migration pattern requires `asyncio.run()` wrapping `connection.run_sync(do_run_migrations)` in `env.py` â€” the synchronous default generated by `alembic init` does not work with an async engine
- The `tsvector` generated column (`GENERATED ALWAYS AS ... STORED`) is not expressible in a SQLModel class; it must be added as raw SQL in the hand-edited migration, not via the model definition. The `search_vector` column is therefore absent from `MedicationRegistry`'s Python class.
- `user_id` in `cabinet_entries` and `user_preferences` is a FK to our local `users.id`, not a direct reference to Supabase's internal `auth.users` table. F-01 will insert into `users` on registration using the UUID assigned by Supabase Auth â€” the UUIDs must match.

## What We're NOT Doing

- Auth routes, JWT validation, or Supabase Auth integration (F-01)
- Registry import script (F-03)
- `dismissed_notifications` table (S-06)
- Any CRUD operations, service logic, or API endpoints â€” those belong to individual feature slices
- `supabase` Python library installation (F-01 concern)
- Row Level Security policies â€” RLS is a safety net; FastAPI enforces isolation as the primary layer

## Implementation Approach

Four sequential phases: install dependencies and create the DB connector first (Phase 1), then define all SQLModel models and scaffold new domain stubs (Phase 2), then initialize Alembic and generate + hand-edit the first migration (Phase 3), then add the smoke test (Phase 4). Each phase leaves the app in a runnable state.

## Critical Implementation Details

**Async Alembic pattern**: The default `env.py` generated by `alembic init` is synchronous. For asyncpg it must be replaced with the async pattern: wrap `do_run_migrations` in `connection.run_sync(...)` inside an `async with engine.connect()` block, and call the whole thing via `asyncio.run()`. Omitting this causes `alembic upgrade head` to hang or fail.

**tsvector generated column**: PostgreSQL's `GENERATED ALWAYS AS (...) STORED` syntax creates a server-computed column. Alembic autogenerate does not emit this; it must be hand-added to the migration as `op.execute(...)` raw SQL after the `create_table` call. The `MedicationRegistry` SQLModel class does not declare `search_vector` â€” the column exists only in the DB.

**Supabase endpoint split (asyncpg + pgbouncer)**: Supabase exposes a direct connection (port 5432) and a transaction-mode pgbouncer pooler (port 6543). asyncpg defaults to server-side prepared statements, which the transaction pooler breaks (`DuplicatePreparedStatementError` / silent failures). Run Alembic migrations against the **direct** connection (5432) â€” DDL must not go through the transaction pooler. The runtime engine may use either; if it uses the pooler, set `statement_cache_size=0` in `connect_args`.

**Explicit `__tablename__` on every model**: SQLModel defaults a table name to the lowercased class name with no snake_casing or pluralization (`MedicationRegistry` â†’ `medicationregistry`, `User` â†’ `user` â€” a PostgreSQL reserved word). Every model MUST declare `__tablename__` (`users`, `user_preferences`, `medication_registry`, `cabinet_entries`) so the autogenerated tables match the FK strings, the hand-edited migration SQL, and the GIN index target.

**dosage_period as CHECK constraint**: `dosage_period` is stored as `VARCHAR` with a `CHECK (dosage_period IN ('day', 'week'))` constraint added in the migration. The SQLModel field is `Optional[str]`; the constraint is hand-added to the migration since autogenerate does not emit CHECK constraints.

## Phase 1: Dependencies and DB Connector

### Overview

Add the three new packages, create the shared DB module, and wire an async lifespan into `main.py` so the engine is created on startup and disposed on shutdown.

### Changes Required

#### 1. Backend dependencies

**File**: `backend/pyproject.toml`

**Intent**: Add `sqlmodel`, `alembic`, `asyncpg`, and `pydantic-settings` as runtime dependencies so the app can connect to PostgreSQL, manage migrations, and load validated config.

**Contract**: Four new entries under `dependencies`: `sqlmodel>=0.0.22`, `alembic>=1.16`, `asyncpg>=0.30`, `pydantic-settings>=2.5`.

#### 2. Settings module

**File**: `backend/app/config.py`

**Intent**: Single validated source of truth for configuration, read once at import. Both the DB connector and Alembic's `env.py` import it so the `DATABASE_URL` is resolved and validated in exactly one place (no raw `os.environ` subscripts scattered around, no `alembic.ini` interpolation).

**Contract**:
- `class Settings(BaseSettings)` from `pydantic_settings`, with `model_config = SettingsConfigDict(env_file=".env", extra="ignore")` so dev picks up a local `.env` automatically.
- Field `database_url: str` (maps to env var `DATABASE_URL`). A missing value raises a clear pydantic `ValidationError` at first access, not a bare `KeyError`.
- Module-level `settings = Settings()` singleton for import by connector and `env.py`.

#### 3. DB connector module

**File**: `backend/app/db/__init__.py`

**Intent**: Mark `db/` as a package so other modules can import from it.

**Contract**: Empty file.

**File**: `backend/app/db/connector.py`

**Intent**: Define the async SQLAlchemy engine, session factory, and a FastAPI dependency (`get_session`) that yields an `AsyncSession` per request. Also expose an `init_db` coroutine used by the lifespan.

**Contract**:
- `engine: AsyncEngine` â€” created from `settings.database_url` (imported from `app.config`) with `create_async_engine`; `echo=False` in production, pool pre-ping enabled. When pointing at Supabase's transaction-mode pooler (port 6543), pass `connect_args={"statement_cache_size": 0}` so asyncpg does not use server-side prepared statements (the transaction pooler breaks them). A direct connection (port 5432) does not require this.
- `async_session_factory` â€” `async_sessionmaker(engine, expire_on_commit=False)`
- `async def get_session() -> AsyncGenerator[AsyncSession, None]` â€” yields one session, closes on exit; used as a FastAPI `Depends`
- `async def init_db() -> None` â€” connectivity check ONLY (e.g. `SELECT 1` against the engine to fail fast on a bad `DATABASE_URL`). Does NOT call `SQLModel.metadata.create_all` â€” Alembic is the sole schema authority. Running `create_all` would build bare tables missing the tsvector column / GIN index / CHECK constraints and then collide with `alembic upgrade head`.

#### 4. FastAPI lifespan

**File**: `backend/app/main.py`

**Intent**: Replace the current argumentless `create_app()` factory with one that registers an async lifespan so the DB engine is initialized on startup and disposed on shutdown.

**Contract**: Add an `@asynccontextmanager async def lifespan(app)` that calls `init_db()` on enter and `engine.dispose()` on exit; pass it to `FastAPI(lifespan=lifespan, ...)`.

### Success Criteria

#### Automated Verification

- `uv run pip check` (or `uv sync`) completes without dependency conflicts
- `uv run uvicorn app.main:app` starts without import errors with a `DATABASE_URL` present in `.env` (a missing value raises a clear pydantic `ValidationError`, not a bare `KeyError`)
- `uv run ruff check . && uv run ruff format --check .` passes

#### Manual Verification

- `backend/app/db/connector.py` exists and is importable
- `backend/app/main.py` references the lifespan

## Phase 3: Alembic Setup and First Migration

### Overview

Initialize Alembic inside `backend/`, configure `env.py` for the async engine and SQLModel metadata, autogenerate the initial migration, then hand-edit it to add the tsvector generated column + GIN index, the `package_count` CHECK constraint, and the `dosage_period` CHECK constraint.

### Changes Required

#### 1. Alembic initialization

**Files**: `backend/alembic.ini`, `backend/migrations/` (created by `alembic init migrations` run from `backend/`)

**Intent**: Bootstrap the Alembic project structure.

**Contract**: Run `uv run alembic init migrations` from `backend/`. This creates `alembic.ini` and `migrations/env.py`, `migrations/script.py.mako`, `migrations/versions/`.

#### 2. Alembic configuration

**File**: `backend/alembic.ini`

**Intent**: Leave `sqlalchemy.url` unset/blank â€” the URL is injected at runtime from the Pydantic settings, not via configparser interpolation (which resolves `%(...)s` from config sections, NOT `os.environ`, and would raise `InterpolationMissingOptionError`).

**Contract**: Remove or blank the `sqlalchemy.url` line; the value is supplied by `env.py`.

**File**: `backend/migrations/env.py`

**Intent**: Replace the default synchronous migration runner with the async pattern required by asyncpg, import all SQLModel models so autogenerate sees the full schema, and inject the DB URL from the shared settings.

**Contract**:
- Import all four model classes (User, UserPreferences, MedicationRegistry, CabinetEntry) before setting `target_metadata`
- Set `target_metadata = SQLModel.metadata`
- Inject the URL from settings: `config.set_main_option("sqlalchemy.url", settings.database_url)` (import `settings` from `app.config`). Use the Supabase **direct** connection (port 5432) for migrations per Migration Notes.
- Replace `run_migrations_online()` with the async pattern:

```python
def run_migrations_online() -> None:
    connectable = create_async_engine(config.get_main_option("sqlalchemy.url"))

    async def run_async_migrations() -> None:
        async with connectable.connect() as connection:
            await connection.run_sync(do_run_migrations)
        await connectable.dispose()

    asyncio.run(run_async_migrations())
```

#### 3. Generate and hand-edit initial migration

**Files**: `backend/migrations/versions/<hash>_initial_schema.py` (generated by `uv run alembic revision --autogenerate -m "initial schema"` from `backend/`)

**Intent**: Capture the full initial schema from SQLModel metadata, then hand-add the three items autogenerate cannot emit.

**Contract** â€” hand-edit additions after the autogenerated `create_table` calls:

1. **tsvector generated column + GIN index** on `medication_registry`:
```sql
ALTER TABLE medication_registry
  ADD COLUMN search_vector tsvector
  GENERATED ALWAYS AS (
    to_tsvector('simple',
      coalesce(name, '') || ' ' || coalesce(active_ingredient, ''))
  ) STORED;

CREATE INDEX ix_medication_registry_search_vector
  ON medication_registry USING GIN (search_vector);
```

2. **CHECK constraint** on `cabinet_entries.package_count`:
```sql
ALTER TABLE cabinet_entries
  ADD CONSTRAINT ck_cabinet_entries_package_count_non_negative
  CHECK (package_count >= 0);
```

3. **CHECK constraint** on `cabinet_entries.dosage_period`:
```sql
ALTER TABLE cabinet_entries
  ADD CONSTRAINT ck_cabinet_entries_dosage_period
  CHECK (dosage_period IN ('day', 'week'));
```

The `downgrade()` function must reverse all three additions.

### Success Criteria

#### Automated Verification

- `uv run alembic upgrade head` completes without error against a real Supabase PostgreSQL DB (requires `DATABASE_URL` set in env)
- `uv run alembic downgrade -1` rolls back cleanly
- `uv run alembic upgrade head` again succeeds (idempotency check)

#### Manual Verification

- Four tables visible in Supabase dashboard: `users`, `user_preferences`, `medication_registry`, `cabinet_entries`
- `medication_registry` has a `search_vector` column of type `tsvector`
- `ix_medication_registry_search_vector` GIN index visible in Supabase â†’ Database â†’ Indexes

## Migration Notes

- The dev `.env` holding `DATABASE_URL` must never be committed (AGENTS.md hard rule); confirm `.env` is gitignored before adding it.
- `alembic downgrade base` drops all tables. Running this against a populated DB will destroy data â€” always back up or use a dedicated test DB.
- The `DATABASE_URL` must use `postgresql+asyncpg://` scheme. A plain `postgresql://` URL will fail with the async engine.
- Run `alembic upgrade head` against the Supabase **direct** connection (port 5432), not the transaction pooler (port 6543) â€” DDL through the transaction pooler fails on asyncpg prepared statements. If the runtime app connects through the pooler, set `connect_args={"statement_cache_size": 0}` on the engine.
